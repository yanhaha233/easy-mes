# 短小精悍的 MES 构建指南

> 本技能用于指导 AI 编码代理（Codex/Claude/Cursor 等）在为中小制造企业开发 MES 系统时，做出"少而精"的设计决策。核心目标是避免大型 MES 产品的复杂度陷阱。

---

## 一、核心理念（最重要，先读这部分）

**MES 的本质只有两件事**：
1. 把生产计划准确传达到工位
2. 把工位上发生的事如实回收回来

所有功能都应围绕这两件事展开。任何让车间工人多花 5 秒以上录入数据的"高级功能"，都要重新评估必要性。

**五条铁律**：
- 移动端优先（车间用平板/手机，不是 PC）
- 离线可用（车间网络不稳是常态，不是异常）
- 手工录入兜底（自动采集是奢侈品）
- 标准产品 + 轻配置（拒绝重度定制）
- 严守边界（不做 ERP 的事：库存、采购、财务、销售订单一律不碰）

---

## 二、反模式：必须主动规避的设计错误

写代码前，先确认你**没有**在做下面这些事：

1. **照搬大型 MES 架构**：不要堆砌 APS 高级排程、SPC 统计过程控制、深度 OEE 分析、完整 WIP 在制品管理。中小厂没有这些数据基础，强上只会变成摆设。

2. **追求全自动数据采集**：老旧设备协议混乱，硬件改造成本可能超过软件本身。**先用扫码 + 手工录入跑通**，关键设备再上传感器。

3. **微服务架构**：中小厂场景下，单体应用完全够用。微服务带来的复杂度远超收益。如果你正在画微服务拆分图，停下来。

4. **强制规范化基础数据**：不要要求客户先把 BOM、工艺路线、标准工时全部规范化才能上线。允许字段缺失、允许 Excel 导入、允许逐步完善。

5. **PC 端为主**：车间工人不会跑到电脑前操作。**所有车间侧功能必须在手机/平板上能用**，PC 端只留管理后台。

6. **报工流程超过 10 秒**：扫码 → 点开工 → 干完点报工 → 录数量 → 选不良原因。整个流程超过 10 秒，工人就会消极对待，数据质量崩盘。这是硬性指标。

7. **过度抽象 / 为未来扩展性留口子**：YAGNI 原则优先。不要为"以后可能要支持多工厂"现在就做多租户分库。

## 三、核心模块（MVP 必备，按优先级排序）

### 3.1 基础档案
- 物料 / 产品（支持单层简化 BOM，多层按需扩展）
- 工艺路线（工序 → 工位 → 标准工时）
- 设备 / 工位
- 人员 / 班组
- **不良原因预设清单**（关键！避免工人打字）

实现要点：
- 所有档案必须支持 Excel 批量导入
- 允许字段缺失，不要做强校验
- 提供"标准工时为空时按经验值估算"的兜底

### 3.2 工单管理
- 工单来源：ERP 同步 + 手工创建（两条路都要支持）
- 自动按工艺路线拆分工序任务
- 状态机：`待开工 / 进行中 / 暂停 / 已完工 / 关闭`
- 字段：优先级、交期、客户、关联销售订单号

### 3.3 车间报工（最高频，必须极致简单）
- 扫工单二维码进入操作页（不要登录跳转）
- 一键开工、暂停、报工
- 报工录入：合格数 + 不良数 + 不良原因（下拉选择，不允许打字）
- **整个流程必须在 10 秒内完成**
- 报工接口必须幂等（断网重传不能产生重复数据）

### 3.4 可视化排产
- 工位横向甘特图
- 拖拽调整工单顺序
- **不要做复杂 APS 算法**，规则越简单越好
- 决策权留给计划员，软件只做信息呈现和约束检查

### 3.5 车间看板
- 大屏展示工单进度、当前产量 vs 计划
- 异常报警（停机超过 N 分钟、不良率超阈值）
- 受众是一线工人和班组长，利用面子心理驱动产出

### 3.6 质量记录
- 首件检验、巡检、终检三类记录
- 不良品台账
- **SPC/Cpk 暂不实现**，客户明确要求再加

### 3.7 简单追溯
- 工单维度：用了哪批物料、过了哪些工位、谁干的、何时干的
- 不强求全条码批次级追溯

### 3.8 基础报表
- 日 / 周 / 月产量
- 简化 OEE = 实际运行时间 / 计划时间（先不做时间损失、性能损失、质量损失三维度拆分）
- 不良率 TOP N 原因
- 人均产出
- 支持 Excel 导出，**不堆 BI 图表**

## 四、明确不做的功能（拒绝清单）

被要求实现以下功能时，先与用户确认必要性，默认不做：

- 复杂 APS 高级排程算法（遗传算法、模拟退火等）
- SPC 统计过程控制（控制图、过程能力指数）
- 完整 WIP 在制品多状态管理
- 深度 PLM / SCADA 集成
- 全自动设备数据采集（先做半自动）
- 库存管理（ERP 的事）
- 采购管理（ERP 的事）
- 财务核算（ERP 的事）
- 销售订单管理（ERP 的事）
- 复杂权限矩阵（先用角色 + 工位的二维矩阵足够）
- 工作流引擎（用状态机就够了，别上 BPMN）

## 五、技术选型基线

除非有充分理由，按下面的选择走：

| 维度 | 推荐 | 不要 |
|------|------|------|
| 架构 | 单体应用 | 微服务 |
| 后端框架 |  FastAPI  | 
| 前端（车间端） | Vue3 + Vite + TypeScript（移动端优先） | 
| 前端（管理端） | Vue3 + Vite + TypeScript + Element Plus | 
| 数据库 | PostgreSQL | 
| 缓存 | Redis | 
| 消息队列 | 一期不要；二期加 RabbitMQ | Kafka（除非真的大数据量） |
| 设备接入 | 一期：扫码 + 手工；二期：OPC UA / Modbus | 私有协议深度集成 |
| 离线方案 | IndexedDB + 后台同步队列 | Service Worker 全量缓存 |
| 部署 | Docker Compose | Kubernetes |
| 网关 / 鉴权 | 单点 JWT | OAuth2 完整体系 |

## 六、数据模型最小集

核心表（按优先级）：

```
materials           # 物料档案
boms                # 简化BOM（material_id, component_id, qty）
routings            # 工艺路线（material_id + version）
routing_operations  # 工序明细（routing_id, seq, work_center_id, std_time）
work_centers        # 工位 / 设备
workers             # 人员
defect_reasons      # 不良原因字典

work_orders         # 工单（主表）
work_order_ops      # 工单工序（按工艺路线展开）
clock_records       # 报工流水（最高频写入）
quality_records     # 质量记录
defect_records      # 不良品台账

trace_links         # 追溯关联（工单 ↔ 物料批次 ↔ 工位 ↔ 人员）
```

设计要点：
- `clock_records` 是最高写入频次的表，要做好索引（work_order_id, work_center_id, created_at）
- 所有表预留 `tenant_id` 字段（SaaS 化必备，但一期可以全为同一个值）
- 时间戳统一用 UTC 存储，前端转本地时区
- 软删除而非物理删除（追溯需要）

## 七、实施路径（项目排期参考）

**第 1 阶段（1-2 个月）**：基础档案 + 工单 + 报工 + 简单看板
跑通"计划 → 执行 → 反馈"主流程，先让数据流转起来。这一阶段不上线排产、报表也行。

**第 2 阶段（1-2 个月）**：可视化排产 + 报表 + 质量记录
让管理层看到数据价值，争取续费/扩单。

**第 3 阶段（按需）**：追溯、设备自动采集、行业模块扩展
客户主动提需求时再加，不要主动推销。

## 八、给 AI 编码代理的硬性约束

每次写代码前自检：

1. **这是核心模块吗？** 不是的话，先问用户确认必要性，不要自作主张开发。
2. **车间端功能考虑了移动端和离线吗？** 没考虑就重写。
3. **报工类高频操作交互超过 10 秒了吗？** 超过就重新设计。
4. **是否在硬编码业务规则？** 不良原因、状态枚举、工序类型这些必须可配置。
5. **是否预留了多租户字段？** SaaS 化必备。
6. **是否在做 ERP 该做的事？** 是的话停下。
7. **是否为"未来扩展性"过度设计？** YAGNI 优先。
8. **API 资源粒度对吗？** 按工单 / 工序 / 工位划分，遵循 REST。
9. **报工接口幂等了吗？** 必须做幂等键，防止断网重传重复写入。
10. **错误处理对车间工人友好吗？** 报错信息要让工人知道下一步该怎么办，不要扔技术堆栈。

## 九、需求识别提示

当用户描述需求时，注意以下关键词，对应映射到本技能的章节：

- "工单 / 派工 / 派单" → §3.2 工单管理
- "报工 / 打卡 / 记产" → §3.3 车间报工（注意 10 秒铁律）
- "排产 / 计划 / 甘特图" → §3.4 可视化排产（拒绝 APS）
- "看板 / 大屏 / 监控" → §3.5 车间看板
- "质量 / 检验 / 不良" → §3.6 质量记录（拒绝 SPC）
- "追溯 / 溯源 / 批次" → §3.7 简单追溯
- "OEE / 稼动率 / 产能分析" → §3.8 基础报表（用简化口径）
- "对接 ERP / 同步订单" → §3.2，注意是单向同步，不要做双向
- "对接设备 / PLC / 数据采集" → 二期再做，先用扫码方案兜底
- "高级排程 / APS / 智能排产" → 拒绝，引导用户接受简单规则排产
- "SPC / Cpk / 过程能力" → 拒绝，引导用户先做基础质量记录
- "工作流 / 审批流" → 用状态机替代，拒绝引入 BPMN 引擎

## 十、终极原则

写完一个功能后，问自己：
- 这个功能能让车间多产出 1 个零件 / 少出 1 个不良吗？
- 这个功能能让计划员少花 10 分钟排产吗？
- 这个功能能让老板看到他想看的数字吗？

三个都答不上，这个功能就不该存在。
